# Audit - WP-2026-152

## What Must Be True
- `_handle_request_changes()` must not accept a generic `IN_PROGRESS` as a valid entry point.
- The handler must accept the requeue only when the latest relevant bus event proves an immediate `REVIEW_DECISION=changes`.
- The handler must derive that signal from the bus events already read in the function and must not perform an extra `latest_event()` bus read.
- The control flow must preserve the existing `UNKNOWN` fallback to execution_log.
- The review bridge must log non-zero `--request-changes` returncodes to stderr without altering the state transition.
- Supervisor dedupe or relaunch policy changes must remain out of scope.
- Tests must exercise the allowed path, the blocked path, the `UNKNOWN` fallback path, and the bridge logging path.

## Failure Modes
1. If the handler still accepts generic `IN_PROGRESS`, the deadlock remains and the ticket is not complete.
2. If the bridge stops logging request-changes failures, the system loses the visibility fix that was already approved.
3. If supervisor dedupe logic is changed in this ticket, scope has drifted and the review must request changes.

## Review Notes
- This ticket is intentionally narrow: it repairs the handoff contract and observability, but does not redesign the review loop.
- The authoritative state source remains the bus.
