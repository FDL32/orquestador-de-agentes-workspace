# AUDIT_WP-2026-150 - Project scanner with import map

## Review Goal
Verify that the scanner produces a compact, deterministic project map with real Python import edges and without pulling noise-heavy directories into the context.

## What Must Be True
- The scanner is stdlib-only and deterministic.
- The emitted map includes a real `importMap` derived from `ast.parse()`.
- Noise-heavy directories are excluded so the map stays compact.
- The output schema is explicit and consumable by the controller without reading scanner internals.
- The artifact lives under `.agent/context/project-map.json` and is read, not recomputed, by the controller.
- The scanner exposes enough structure for the controller to enrich `## Project Context`.
- The implementation remains fast enough to run during normal ticket prep.
- `SyntaxError` parse failures are represented in the output.

## Failure Modes
1. The scanner reintroduces noise-heavy directories and makes the output too large.
2. The scanner infers imports with string matching instead of AST parsing.
3. The output is not deterministic across repeated runs on the same tree.
4. The output schema is ambiguous, making the controller integration brittle.
5. The controller hook duplicates scanner logic instead of consuming its summary.
6. `SyntaxError` parse failures disappear silently instead of being reported.
7. Tests only cover the happy path and miss exclusion, import discovery, parse-failure, or determinism regressions.

## Review Checklist
- Confirm the scanner output is compact and token-efficient.
- Confirm Python import edges are produced from AST parsing.
- Confirm excluded directories do not contaminate the map.
- Confirm the controller reads `.agent/context/project-map.json` rather than re-scanning inline.
- Confirm `SyntaxError` failures are surfaced in the output.
- Confirm the controller integration consumes the scanner summary rather than reimplementing it.
- Confirm tests directly exercise the scanner behavior.
