# PLAN_WP-2026-150 - Project scanner with import map

## Goal
Build a deterministic, stdlib-only scanner that turns a destination repository into a compact `project-map.json` with file inventory, category buckets, fingerprints, framework hints, and a real Python `importMap`.

## Strategy
1. Scan the destination tree deterministically with `pathlib` and `ast.parse()`.
2. Exclude noise-heavy directories so the resulting map stays token-efficient and relevant to the Builder.
3. Capture Python import relationships from AST instead of heuristics, preserving full module paths for local imports and collapsing only stdlib/external imports to top-level names.
4. Detect framework hints from common manifests.
5. Keep the output compact enough to seed `## Project Context` in the ticket flow.
6. Write the artifact to `.agent/context/project-map.json` and let the controller consume it instead of re-scanning inline.
7. Add tests for exclusions, import discovery, fingerprints, framework hints, parse failures, and deterministic output.

## Scope
- `scripts/project_scanner.py`
- `.agent/agent_controller.py`
- `tests/unit/test_project_scanner.py`

## Acceptance Shape
- The scanner emits a compact `project-map.json`.
- The output includes a real `importMap` for Python files.
- The output excludes noise-heavy directories and stays token-efficient.
- The output is written under `.agent/context/` and can be consumed by the controller without inline rescans.
- The controller can consume the scanner summary to enrich `## Project Context`.
- The scanner is deterministic and uses only the Python stdlib.
- Tests cover the main scan path, the exclusion path, and parse failures.

## Notes
- No new dependencies are allowed.
- The scanner is a data-shaping layer, not a graph UI or an LLM-powered analyzer.
- This ticket should remain focused on compact project understanding for destination repos.
- `SyntaxError` must be reported explicitly in the output instead of disappearing into a broad exception handler.
