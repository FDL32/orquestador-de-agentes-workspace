## Evidencia de Fase 0 (read-only, sin diff productivo)

| Modo | Run | Resultado | Tests rojos |

- Bajo loadscope el conteo oscila 1<->3 y el conjunto cambia entre corridas: NO es estable.
- Control serial decisivo: los 3 tests implicados **pasan serial** (`3 passed in 188s`). El defecto es
  exclusivamente de paralelizacion, no de los tests.

## Causa raiz (por que loadscope no basta)

- Los tests no parallel-safe persistentes son los 3 que `011e` ya habia documentado:
  `test_upgrade_path_suggestion` (detect_version), `test_scan_current_project` (project_scanner),
  `test_no_inline_ticket_regex`. Dependen de estado GLOBAL del proceso/repo real (cwd, git del repo
  actual, escaneo del proyecto vivo), no de estado agrupable por archivo.
- `--dist loadscope` agrupa tests por modulo en el mismo worker; elimina la contencion CROSS-archivo
  (la de project_root_resolution que vimos en 013b) pero NO la contencion sobre recursos globales del
  proceso compartidos por TODOS los workers. Por eso el rojo se reduce pero no desaparece.

## Por que bloquea (limites del contrato)

- La unica via verde estable exigiria una de estas, todas fuera del contrato actual:
  1. Reescribir esos 3 tests para no depender de cwd/git/escaneo global (codigo de tests de producto,
     y ya se vio en 013b que "reescribir tests" no es el patron correcto aqui; ademas no es la superficie
     declarada de 011i, que es el runner).
  2. Marcar esos 3 tests para forzarlos a un worker dedicado / serial dentro del subset paralelo
     (requiere `@pytest.mark` + logica de seleccion en el runner: nueva interfaz, no contemplada).
  3. Excluir esos 3 del subset por defecto paralelo (cambia que se ejecuta por defecto: riesgo de cobertura
     silenciosa; el contrato prohibe "interfaz de opt-out fuera del CLI actual").
- Cualquiera amplia el scope mas alla de "activar default + loadscope en el runner".

## Recomendacion (no ejecutada; decision del humano/Manager)

El default xdist para `--level unit` NO es alcanzable solo con politica de reparto. Opciones reales:
1. **Aceptar el opt-in actual como estado final** (011e + piloto CI de 010m) y NO promover xdist a default.
   El usuario YA tiene paralelizacion bajo demanda; el default automatico no es seguro sin mas trabajo.
2. **Abrir un ticket nuevo y honesto de "3 tests global-state-bound"**: hacer esos 3 tests robustos a
   ejecucion paralela (fixtures que aislen cwd/git/escaneo), como prerequisito real del default. Es trabajo
   de tests de producto, no de runner, y debe contratarse como tal con su propia evidencia.
3. **Default xdist con esos 3 forzados a serial** via marca + soporte en el runner: requiere disenar esa
   interfaz; es 011i ampliado, no el 011i actual.
