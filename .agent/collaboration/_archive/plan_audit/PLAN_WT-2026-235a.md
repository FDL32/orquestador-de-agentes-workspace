# PLAN_WT-2026-235a

## 1. Contrato Tecnico

El Manager review bridge no puede convertir texto ambiguo del transcript en una
decision accionable. Una decision accionable es aquella que cambia el flujo:

- `APPROVE`: permite cierre canonico.
- `CHANGES`: requeue hacia Builder.

Ambas requieren procedencia autoritativa. Para este ticket, la fuente autoritativa
minima es `json_final_answer`. El fallback `text_regex` es diagnostico, no autoridad.

## 3. Cambio Esperado

### Procedencia

Actualmente `_parse_opencode_decision()` puede caer a `text_regex` y buscar:

```python
DECISION:\s*CHANGES
DECISION:\s*APPROVE
```

en todo el `stdout`. Eso es peligroso porque el propio review packet contiene esas
cadenas como instrucciones.

Nuevo contrato:

- `json_final_answer` puede producir `APPROVE`, `CHANGES` o `INSPECT`.
- `json_last_text` no es autoritativo: nunca puede producir `APPROVE` ni
  `CHANGES`; degrada a `INSPECT`.
- `text_regex` no puede producir decision fuerte. Si encuentra patrones en stdout
  completo, devolver `INSPECT` con `parse_method=text_regex` o equivalente
  degradado.

### Payload CHANGES

Usar `_validate_changes_structure()` como gate real:

- Si falta `## SUMMARY`, `## BLOCKERS`, `## SUGGESTIONS` o `DECISION: CHANGES`,
  `decision` final debe ser `INSPECT`.
- Si `blockers.strip()` esta vacio, `decision` final debe ser `INSPECT`.
- El payload debe incluir:
  - `decision`
  - `stdout_tail`
  - `blockers`
  - `parse_method`
  - `failure_reason` cuando haya degradacion
  - `missing_sections` cuando aplique

## 4. Tests Esperados

1. `test_text_regex_template_decisions_are_inspect`
   - stdout contiene plantilla con `DECISION: APPROVE` y `DECISION: CHANGES`.
   - No hay veredicto final autoritativo.
   - Esperado: `INSPECT`, no `APPROVE` ni `CHANGES`.

2. `test_tool_calls_without_final_answer_is_inspect`
   - NDJSON con un evento `type=text`, `phase != final_answer`, que contiene
     `DECISION: CHANGES`.
   - El stream termina en `step_finish`/`tool-calls`, sin `phase=final_answer`.
   - Esperado: `INSPECT`.

3. `test_changes_without_blockers_degrades_to_inspect`
   - Decision inicial `CHANGES`, pero sin bloque `## BLOCKERS` o con blockers vacio.
   - Esperado: `INSPECT`, `failure_reason=changes_structure_invalid`.

4. `test_structured_changes_preserves_blockers`
   - Respuesta final con `## SUMMARY`, `## BLOCKERS`, `## SUGGESTIONS`,
     `DECISION: CHANGES`.
   - Esperado: `CHANGES` y `payload.blockers` no vacio.

5. `test_authoritative_approve_still_approves`
   - Respuesta final autoritativa con `DECISION: APPROVE`.
   - Esperado: `APPROVE`.

6. `test_review_decision_payload_records_parse_method`
   - Al emitir `REVIEW_DECISION`, el payload incluye `parse_method`.
