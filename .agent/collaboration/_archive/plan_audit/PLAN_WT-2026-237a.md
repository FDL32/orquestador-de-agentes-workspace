## 3. Cambio Esperado

### Scope canonico

Todo cambio nuevo del `repo_motor` debe quedar dentro de `Files Likely Touched`
del ticket. Si un comportamiento ya esta correcto en `main` y no necesita ajuste
ni test adicional, no volver a tocarlo solo para "reformalizarlo".

### Hardening residual

Resolver solo gaps que sigan abiertos tras leer el codigo real. Prioridad:

1. transporte/review bridge cuando la salida de OpenCode introduce ruido benigno
   o errores reales de auth;
2. closeout/mark-ready para tickets no-code y mixed;
3. proyeccion de estado y reconciliacion cuando el supervisor no materializa;
4. launcher PowerShell con `Set-StrictMode` y permisos runtime.

### Packaging de ticket code

- `execution_log.md` debe registrar gates reales con exit code.
- El commit final debe mencionar exactamente `WT-2026-237a`.
