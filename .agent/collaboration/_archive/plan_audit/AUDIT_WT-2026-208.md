# AUDIT WT-2026-208 - Estabilizacion de suite global

Alias: Suite Stabilization / Testing Contract Audit
Validacion: externa al loop Builder/Manager; no usar solo el contador final de pytest como criterio de verdad.

## Objetivo del audit
Verificar que `WT-2026-208` corrige la suite global de forma estructural y secuencial,
no con parches rapidos que oculten deuda.

## Reglas de revision
- No aprobar por una simple bajada del numero de fallos.
- Exigir evidencia por familias y por pasadas.
- Bloquear cualquier fix que reintroduzca topologia legacy para contentar tests.
- Bloquear cualquier cierre con skips/xfails amplios no justificados.

## Hallazgos bloqueantes tipicos
1. **Parche local sin contrato**
   - El Builder cambia tests o codigo para que pasen, pero no explica si el contrato vigente es nuevo o legacy.

2. **Reintroduccion de rutas antiguas**
   - Aparecen `z_scripts`, repo anidado o paths hardcodeados que contradicen workspace+motor.

3. **Manipulacion del repo vivo en tests**
   - Los tests siguen renombrando/ocultando `.agent` real o dependen de permisos frágiles de Windows.

4. **Residuals sin clasificar**
   - Quedan fallos al final y el Builder no los separa entre regresion, test obsoleto o deuda nueva.

5. **Encoding degradado**
   - Se corrigen tests de mojibake degradando superficies operativas vigentes o metiendo mas ruido de encoding.

## Evidencia minima esperada
- baseline inicial con recuento y familias;
- una narrativa de pasadas (familia -> fix -> rerun -> resultado);
- reruns de subconjuntos por familia;
- rerun final de la suite completa;
- justificacion explicita de cada cambio de test sensible.

## Checklist TP
- TP-01: el Builder presento baseline agrupada por familias.
- TP-02: al menos una familia quedo completamente cerrada con evidencia.
- TP-03: no hay rutas legacy nuevas ni supuestos de repo anidado.
- TP-04: los tests que tocaban `.agent` real ahora usan estrategia segura o justificada.
- TP-05: las correcciones de semantica distinguen bien entre deuda legacy y regresion real.
- TP-06: la suite completa fue reruneada al final.
- TP-07: los residuals, si quedan, estan clasificados y empujan backlog claro.

## Verificacion recomendada
1. Leer diff por familias, no solo por archivo.
2. Ejecutar subconjuntos criticos de cada pasada.
3. Ejecutar la suite completa del motor.
4. Revisar que `execution_log.md` documente baseline, pasadas y residuals.

## Criterio de rechazo inmediato
- `READY_FOR_REVIEW` sin rerun final de la suite completa.
- Uso de `skip`/`xfail` como mecanismo principal de cierre.
- Introduccion de paths `z_scripts` o hardcodes incompatibles con workspace+motor.
- Cualquier fix que tape el problema en tests pero empeore el contrato del sistema.
