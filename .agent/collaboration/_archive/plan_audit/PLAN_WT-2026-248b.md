# PLAN_WT-2026-248b - Contract hardening prompt <-> skill

## Resumen ejecutivo
`WT-2026-248b` endurece el contrato entre prompts wrapper y skills canonicas
del motor. El ticket existe porque hoy hay drift verificable: prompts de rol
que no nombran su skill fuente y skills con workflow, gates o vocabulario que
ya no representan el contrato real del bus.

El deliverable es `mixed`: cambia prompts, skills, una barrera ejecutable y la
documentacion canonica. No incluye cambios a `.opencode/opencode.json` ni al
runtime temporal de OpenCode.

## Contrato canonico protegido
- La skill es la fuente canonica del workflow, constraints y references.
- El prompt es un wrapper contextual que debe apuntar explicitamente a la skill.
- El contrato debe ser bidireccional y verificable:
  - skill -> `source_prompt:` + `contract_id` en frontmatter;
  - prompt -> linea visible `Skill canonica: <ruta_skill>` + mismo `contract_id`.
- La identidad contractual no puede descansar en frases naturales fragiles; el
  `contract_id` es la prueba principal y los `rg` quedan como limpieza
  auxiliar de deuda heredada.
- La barrera de regresion no puede depender de auto-reporte; debe ejecutarse
  como comando verificable en gates.

- cobertura en `tests/test_discover_skills.py`
- integracion en `scripts/run_gates_dispatch.py`
- `PROJECT.md` y `CHANGELOG.md` actualizados

Evidencia:
- checker: exit 0 sin warnings
- test: verde
- ruff: verde
- gates aplicables: checker ejecutado

Implementacion obligatoria de Fase 3:
- `extract_frontmatter()` o una variante equivalente debe distinguir entre
  "sin frontmatter", "YAML invalido" y "frontmatter valido"; YAML invalido no
  puede colapsar silenciosamente a `{}`.
- El checker debe exigir `contract_id` en skills de `role: manager|builder` y
  verificar que el prompt referenciado declara exactamente el mismo valor.
- El checker debe resolver `source_prompt:` relativo al `repo_motor`; rutas
  absolutas o relativas al `repo_destino` deben fallar como no portables.
- `run_gates_dispatch.py` debe invocar `python scripts/discover_skills.py --check-contract`
  desde `main()` como paso final e independiente del `deliverable_type`; exit
  distinto de 0 debe fallar el gate.

## Riesgos
- falso positivo si el checker no verifica ancla inversa
- falso positivo si el checker acepta `source_prompt:` portable solo en el
  entorno local del Builder
- falsa seguridad si el checker no entra en gates
- `grep` demasiado fragil si se usa como contrato principal
- frontmatter YAML invalido confundido con contrato invalido
