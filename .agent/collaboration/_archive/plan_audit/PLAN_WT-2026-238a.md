## 1. Contrato Tecnico

Este ticket existe para cerrar la sesion de forma deliberada y dejar un handoff
limpio para el siguiente chat. No es un ticket de codigo ni una reapertura de
`WT-2026-237a`.

El output minimo es:

  falta promover;

  adicional;
- handoff corto y canonico para arrancar el siguiente chat sin reanalizar toda
  la sesion;
- `validate --json` del `repo_destino` sin errores.

## 1b. Contexto y Problema

La sesion que cerro `WT-2026-237a` produjo cambios utiles en varios planos:

- memoria promovida al `repo_motor`;
- documentacion durable de descubribilidad (`AGENTS.md`,
  `REPOSITORY_STRUCTURE.md`, `docs/KNOWN_FAILURE_PATTERNS.md`);
- cierre canonico del ticket de codigo en el `repo_destino`.

Si el siguiente chat arranca sin un handoff compacto, el sistema seguira siendo
correcto, pero el contexto util quedara disperso entre backlog, memoria,
execution log y cambios recientes del `repo_motor`.

## 1c. Decision Arquitectonica

Este ticket adopta el principio de "contexto bajo demanda":

- no inflar prompts core del Builder o del Manager con arquitectura fija;
- no usar memoria operativa como sustituto de documentacion durable;
- dejar un paquete pequeno y claro de cierre de sesion que apunte a las fuentes
  correctas.

El objetivo no es documentar todo otra vez, sino dejar claro:

1. que quedo resuelto;
2. que se promociono a memoria o documentacion durable;
3. que queda pendiente para el siguiente ciclo.
