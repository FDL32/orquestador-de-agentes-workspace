# PLAN_WT-2026-230a

## Objetivo
Dar al motor un arranque canonico y ligero para `repo_destino` local: un mapa compacto
del destino generado desde `repo_motor`, sin Node/Repomix/Graphify obligatorios y sin
duplicar logica ejecutable en cada destino.

## Root cause verificada
- `install_agent_system.py` ya provisiona y sincroniza estructura, config y templates,
  pero no existe una capacidad especifica de "entender rapido el destino" para agentes nuevos.
- `session_bootstrap.md` es util para el motor, pero no resuelve por si solo el caso
  `destination-hosted`.
- Repomix como contexto completo del motor ya mostro dos limites: permisos/globby en
  Windows y volumen excesivo de contexto para arranque general.
- En repos locales, `rg` y lectura directa son baratos; lo que falta no es profundidad
  automatica sino una orientacion inicial compacta y determinista.

## Diseno aprobado

### A. Logica en el motor
Nuevo archivo:
- `scripts/destination_context.py`

Propiedades:
- stdlib solamente
- invocado siempre desde `repo_motor`
- recibe `--project-root`
- genera `destination_map.md` dentro del destino
- no depende de Graphify, Git o Repomix para funcionar

### B. Datos/config en el destino
Provisionados por el instalador:
- `.agent/context/`
- `.agent/config/destination_context.json`

No se copia al destino:
- `destination_context.py`
- wrappers `.ps1/.bat/.sh`
- ninguna logica ejecutable equivalente

### C. Prompt de arranque
Nuevo prompt:
- `prompts/destination_bootstrap.md`

Secuencia canonica del prompt:
1. leer `.agent/config/motor_destination_link.json`
2. extraer `motor_root`
3. ejecutar `python <motor_root>/scripts/destination_context.py --bootstrap --project-root <repo_destino>`
4. leer `.agent/context/destination_map.md`
5. continuar con `rg` y lectura directa

Nota de implementacion: esa secuencia es para agentes que arranquen despues en un
`repo_destino`. El Builder de este ticket no debe ejecutarla contra el destino real.
Para verificar el contrato del link, lee `write_motor_destination_link()` y
`resolve_motor_root()` en `repo_motor`; para pruebas, usa destinos simulados en
`tmp_path`.

Actualizacion a `session_bootstrap.md`:
- declarar modo `motor-only`
- redirigir a `destination_bootstrap.md` cuando el agente opere sobre `repo_destino`

## Contenido minimo de `destination_map.md`

### 1. Identidad y topologia (protegida)
- ruta absoluta del destino
- ruta del motor
- modo `destination-hosted`
- `Ticket prefix` si existe
- indicacion de si el link motor-destino es valido

### 2. Estado operativo (protegido)
- ticket activo si existe
- rol/accion actuales si existen
- referencias a `work_plan.md`, `TURN.md`, `STATE.md`
- `PLAN_*.md` / `AUDIT_*.md` activos si aplican

### 3. Estado git (best effort)
- branch
- HEAD corto
- working tree limpio/sucio
- lista corta de archivos modificados

### 4. Estructura del repo (truncable)
- arbol filtrado y de profundidad acotada
- excluir `.git`, caches, builds, binarios, `.agent/context`

### 5. Contratos y manifests (truncables)
- extractos compactos de `AGENTS.md`, `PROJECT.md`, `README*`
- `pyproject.toml`, `package.json` o equivalentes si existen

### 6. Resumen opcional (truncable)
- graphify solo si ya existe

## Presupuesto de bytes
- Config por defecto: `204800` bytes (`200 KB`)
- El generador construye secciones en memoria y solo serializa al final
- Prioridad de conservacion:
  1. identidad/topologia
  2. estado operativo
  3. estado git
  4. contratos/manifests
  5. arbol
  6. graphify opcional
- Si falta espacio:
  - primero reducir graphify
  - luego recortar arbol
  - luego truncar extractos secundarios
  - nunca perder identidad + estado operativo

## Comportamiento de errores

### `motor_destination_link.json` ausente o invalido
- error claro
- exit code != 0
- no escribir mapa engañoso

### Git ausente o repo no versionado
- no crash
- seccion git degradada con nota explicita

### Graphify ausente
- no crash
- simplemente no incluir resumen graphify

### Archivos opcionales ausentes
- no crash
- reflejar ausencia en el mapa o resumen final

## Provisionado del instalador
Seguir el patron existente `copy_X(...)`:
- nueva funcion `copy_destination_bootstrap(...)` o equivalente
- llamada desde `install_agent_system()` y `sync_agent_system()`
  (orquestadas por `main()` via `args.install` / `args.sync`)

Reglas:
- crea `.agent/context/` si no existe
- deposita `destination_context.json` si no existe
- no imita el overwrite incondicional de `copy_repomix_config`; debe usar guard
  `if not exists` o el mecanismo canonico de rutas depositadas una vez y luego
  poseidas por el destino, de forma consistente con el contrato real del instalador
- no pisa personalizaciones locales
- mantiene idempotencia

## Tests esperados

3. repo no versionado no hace crash

## Deuda explicitamente diferida
- skill de bootstrap de destino
- cache del mapa
- perfiles por dominio/capas
- subtree-pack con Repomix
- extender `--validate` con semantica de descubrimiento
- wrappers locales en el destino

## Barrera operativa para Builder
- Builder no lee ni escribe ningun path real bajo `repo_destino` / `workspace_activo`,
  incluyendo `.agent/config/`, `.agent/collaboration/`, `.agent/runtime/` y `backlog.md`.
- La implementacion productiva vive en `repo_motor`; el repo_destino solo refleja
  estado y artefactos operativos manejados por Manager/supervisor.
- Si el Builder necesita datos del destino, debe obtenerlos desde el codigo generador
  en `repo_motor` o desde fixtures temporales creadas por los tests.
