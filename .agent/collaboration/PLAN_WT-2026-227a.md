# PLAN WT-2026-227a

## Objetivo
Hacer observable el resultado de Repomix dentro del contexto de review del
Manager, sin convertirlo en gate obligatorio.

## Scope minimo
- tocar `bus/review_bridge.py:_ensure_repomix_context`;
- actualizar `bus/review_bridge.py:_run_opencode_review` si cambia la firma de
  `_ensure_repomix_context`;
- exponer `repomix_status` en el review context o estructura equivalente;
- capturar exito, returncode/stderr y excepciones de subprocess;
- probar que el review continua cuando Repomix falla.

## Seams reales confirmados
- `bus/review_bridge.py:_ensure_repomix_context`
- `bus/review_bridge.py:_run_opencode_review`, caller que hoy usa
  `repomix_path = self._ensure_repomix_context(); if repomix_path:`
- `tests/test_manager_review_bridge.py:_mock_repomix_for_tests`

## Contrato de entrega
1. `repomix_status.status` toma los literales exactos `"ok"`, `"failed"` o
   `"skipped"`;
2. fallos de subprocess exponen `reason`, `returncode` y `stderr_tail` cuando
   existan;
3. excepciones como `FileNotFoundError` quedan capturadas con razon verificable;
4. Repomix sigue siendo best-effort;
5. los tests focales nuevos no usan `_mock_repomix_for_tests`.

## Estrategia de firma
Opcion recomendada: `_ensure_repomix_context` devuelve
`(Path | None, repomix_status_dict)` y `_run_opencode_review` desempaqueta antes
de decidir si agrega `-f <path>`. No devolver solo un dict desde
`_ensure_repomix_context`.

## Tests esperados
- exito Repomix -> `status = ok`;
- returncode distinto de 0 -> `status = failed`, `returncode`, `stderr_tail`;
- excepcion de subprocess -> `status = skipped` o `failed` con `reason`;
- review continua pese a fallo Repomix.
