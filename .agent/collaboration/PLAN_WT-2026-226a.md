# PLAN WT-2026-226a

## Objetivo
Unificar la evidencia usada por `--mark-ready` y por el review packet para que
ambos vean los mismos archivos/commits productivos del ticket en `repo_motor`.

## Scope minimo
- diagnosticar las dos rutas reales de evidencia;
- evitar que un working tree sucio oculte commits validos del ticket;
- hacer que mark-ready y review packet consuman un criterio compartido;
- cubrir el caso `commit del ticket + working tree sucio` con tests.

## Seams reales confirmados
- `.agent/agent_controller.py:_check_implementation_evidence`
- `.agent/agent_controller.py:_collect_git_diff_files`
- `bus/review_bridge.py:_get_motor_diff_files`
- `bus/review_bridge.py:classify_review_packet`
- `repo_motor` como repositorio git canonico del motor
- seam compartido en nuevo modulo `bus/evidence.py`, nunca bajo `.agent/` ni
  `bus/utils.py`

## Contrato de entrega
1. `mark-ready` y review packet deben coincidir ante el mismo estado de
   evidencia;
2. commits recientes del ticket en `repo_motor` se incluyen de forma
   incondicional en la evidencia, no solo como fallback;
3. un working tree sucio no relacionado no puede producir `empty review diff`
   con commit valido del ticket presente;
4. el caso sin diff ni commit verificable sigue bloqueado/rechazado;
5. no se crea un segundo gate paralelo ni se relaja el gate existente.

## Decision de ubicacion
El helper compartido debe vivir en `bus/evidence.py`, importable por
`.agent/agent_controller.py` y por `bus/review_bridge.py`. No alojarlo en
`.agent/`, porque el bridge no importa el controller ni debe subprocesarlo para
resolver evidencia. No usar `bus/utils.py`: ese modulo esta reservado a helpers
puros sin estado ni I/O.

## Forma de datos requerida
La evidencia estructurada debe exponer:
- `motor_files` y `destination_files` separados para el bridge;
- archivos de working tree y de commits recientes del ticket;
- veredicto combinado `has_productive_evidence` para el controller;
- resumen de fuentes usadas.

## Diagnostico obligatorio
- registrar que el reason string de `WT-2026-225a` apunta al camino diff-seam
  (`empty review diff`) como inferencia razonable; el working tree historico no
  es recuperable;
- cubrir el camino diff-seam;
- registrar `bus-context-null` en `classify_review_packet` como deuda explicita
  si no entra en scope.

## Tests esperados
- test de regresion: commit `WT-2026-226a` en `repo_motor` + working tree sucio
  no relacionado -> review packet no queda empty;
- usar repos git reales en `tmp_path` siguiendo el patron `init_git_repo` de
  `tests/test_pre_handoff_guard.py`; no mockear subprocess de git;
- test de paridad semantica: consumidor de `mark-ready` y consumidor de review
  packet reportan el mismo veredicto `has_productive_evidence` para el fixture;
- test negativo: sin commit del ticket y sin diff productivo -> bloqueo o
  clasificacion empty;
- test de no scope creep: no tocar rounds, locks ni relaunch.
- los tests se anaden a `tests/test_agent_controller.py` y
  `tests/test_review_bridge.py`; no crear archivos paralelos para este ticket.

## Evidencia minima
- tests focales verdes;
- `ruff check` verde;
- `validate --json` del `repo_destino` en `0/0`;
- execution_log con comandos exactos y resultados.
