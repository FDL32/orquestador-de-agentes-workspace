# PLAN_WT-2026-232a

## Objetivo
Hacer motor-aware el scope gate de `--mark-ready` en la topologia actual
`repo_motor` + `repo_destino`, eliminando el `--scope-override` normal para entregas
correctas dentro de `Files Likely Touched`.

## Non-goals
- No modificar `bus/evidence.py` ni ningun archivo bajo `bus/`.
- No implementar la resolucion de raices en `runtime/motor_link.py`.
- No modificar globalmente `check_scope_gate()`.
- No duplicar en PowerShell la logica Python de resolucion de topologia.

En el mismo ciclo, normalizar las superficies operativas que aun hablan de `Modelo B`
y dejar el arranque Builder portable: FLT persistente motor-relative, root local del
motor solo como dato efimero de prompt/runtime.

## Causa Raiz Verificada
- `WT-2026-231a` corrigio `--pre-handoff`: commit y tag se hacen en `repo_motor`.
- `--mark-ready` conserva el gate legacy:
  - `parse_files_likely_touched()` resuelve FLT contra `PROJECT_ROOT`;
  - `get_changed_files()` mira `repo_destino` cuando este tiene `.git`;
  - el check best-effort puede emitir `No Files Likely Touched match git changes`
    aunque exista commit real en `repo_motor`.
- El cierre de `WT-2026-231a` necesito `--scope-override` pese a tener commit motor
  correcto (`2a0d784`) y checkpoint.

## Diseno

### 1. Separar scope legacy de scope motor-aware
El path de `mark-ready` debe distinguir:
- evidencia productiva en `repo_motor`;
- cambios de estado/colaboracion en `repo_destino`;
- fallback legacy para destinos donde no exista link/topologia motor.

### 2. FLT raw motor-relative
No usar `parse_files_likely_touched()` para comparar contra `repo_motor`, porque
resuelve contra `PROJECT_ROOT`.

Reusar `_parse_raw_flt_paths()` ya definida en `.agent/agent_controller.py`.
No crear una funcion duplicada.

Contrato:
- entrada: contenido de `work_plan.md`;
- salida: paths relativos a `repo_motor` con `/`;
- no resolver con `Path.resolve()`.

### 3. Evidencia motor-aware
`mark-ready` NO debe llamar a `resolve_evidence()` para resolver el scope motor-aware.

Implementar en `.agent/agent_controller.py`:
`_resolve_motor_checkpoint_files(motor_root: Path, ticket_id: str) ->
tuple[bool, set[str]]`.

El helper ejecuta `git rev-parse checkpoint/review-<ticket_id>`, comprueba con
`git merge-base --is-ancestor <sha> HEAD` que el SHA es alcanzable, verifica que el
mensaje contiene `ticket_id` y obtiene los archivos mediante
`git diff-tree --no-commit-id -r --name-only <sha>`.

Si existe topologia motor/destino y cualquier comprobacion falla, `mark-ready`
bloquea por falta de evidencia. El fallback legacy solo se usa cuando no existe
topologia motor/destino.

No basta encontrar cualquier commit historico con el mismo ID dentro de una ventana
de log: la evidencia de scope pertenece al checkpoint de la ronda actual.

Si el commit motor toca al menos un FLT y no toca paths fuera de FLT, el scope gate
pasa.

Punto de inyeccion exacto: `_handle_mark_ready()`, en el bloque que hoy llama a
`check_scope_gate()`. Añadir la rama motor-aware antes de esa llamada y conservar el
fallback legacy. No modificar `check_scope_gate()` directamente.

### 4. Bloqueos
- Sin evidencia motor y sin cambios productivos: bloquear como hoy por no evidence.
- Evidencia motor fuera de FLT: bloquear con lista motor-relative exacta.
- Evidencia solo de colaboracion del destino: bloquear como collaboration-only.

### 5. Terminologia y arranque portable
- Cambiar referencias operativas a `Modelo B` / `Model B` por `topologia motor/destino`.
- El alcance de "superficies operativas" para esta migracion de terminologia es:
  - Archivos del FLT declarado: prompts, `.opencode/agents/builder.md`,
    `scripts/launch_agent_terminals.ps1`, `.agent/agent_controller.py`.
  - No incluye: docstrings internos de `runtime/motor_link.py`, `bus/supervisor.py`,
    `scripts/prepush_check.py`, `scripts/session_closeout.py`; ni `AGENTS.md` del motor;
    ni `prompts/audit_plan.md`; ni `CHANGELOG.md`; ni referencias en tickets cerrados.
  - Esas menciones quedan como "legacy marcadas" sin tocarlas.
- El launcher debe inyectar tres variables efimeras exactas en el prompt Builder:
  `repo_motor_root`, `repo_destino_root` y `workspace_activo_root`.
- Implementar `_resolve_launcher_roots(project_root: Path) -> dict[str, str]` en
  `.agent/agent_controller.py` como fuente unica de esas tres claves exactas.
- Exponerlo mediante
  `--resolve-launcher-roots --json --project-root <repo_destino>`.
- `scripts/launch_agent_terminals.ps1` debe consumir esa salida JSON y no duplicar la
  resolucion de topologia.
- `repo_destino_root` se resuelve desde `-ProjectRoot`; `repo_motor_root`, desde
  `.agent/config/motor_destination_link.json` con fallback a la raiz derivada de la
  ubicacion del launcher; y `workspace_activo_root`, desde la raiz operativa que
  contiene `.agent/`.
- Precedencia: argumento/runtime explicito, enlace valido y fallback derivado.
  Los tres valores deben ser no vacios y normalizados. Si no pueden resolverse de
  forma coherente, el launcher bloquea con un mensaje accionable.
- `TURN.md`, `work_plan.md` y `PLAN_*.md` mantienen paths motor-relative.
- No persistir rutas absolutas locales como contrato.
