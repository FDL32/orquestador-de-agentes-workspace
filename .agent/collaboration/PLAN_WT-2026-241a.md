# PLAN_WT-2026-241a

## 1. Contrato Tecnico

Este ticket endurece el arranque del Builder para que el launcher limpie
procesos Builder previos del mismo `project_root` antes de lanzar una nueva
ronda. El objetivo es evitar shells stale que luego emiten
`HANDOFF_BLOCKED stale_builder_round`.

El output minimo es:

- deteccion de procesos Builder previos del proyecto;
- cierre de esos procesos antes del nuevo launch;
- limpieza asociada de `builder_lock.txt` y `builder_session.json` si hubo
  procesos Builder previos;
- cobertura focal del hardening.

## 2. Superficie

### Builder

- `scripts/launch_agent_terminals.ps1`

### No tocar salvo evidencia nueva

- `.agent/agent_controller.py`
- `bus/supervisor.py`
- `tests/test_pre_handoff_multirepo.py`

## 3. Cambio Esperado

### Cambio minimo de motor

- `Stop-ProjectAgentProcesses` debe distinguir un modo de limpieza orientado a
  Builder.
- En ese modo debe reconocer procesos Builder del mismo `project_root`,
  incluyendo señales de OpenCode / variables de ronda del Builder.
- Antes de `LaunchBuilder`, el launcher debe ejecutar esa limpieza especifica.
- La limpieza Builder debe añadirse tambien en la ruta de `ResumeBuilder`,
  antes de que el launcher llegue al bloque efectivo de `LaunchBuilder`. No
  basta con la limpieza general del arranque fresco.
- Si se cerraron procesos Builder previos:
  - eliminar `builder_lock.txt` si existe;
  - eliminar `builder_session.json` si existe, pero solo despues de haber
    cerrado procesos Builder previos y antes de lanzar la nueva ronda;
  - dejar trazabilidad por stdout del launcher.

### Resultado observable

- relanzar Builder con ventanas previas abiertas ya no deja shells viejas del
  mismo ticket intentando cerrar;
- el siguiente Builder arranca con lock y session limpios;
- disminuyen o desaparecen `stale_builder_round` inmediatamente posteriores al
  launch.

## 4. Tests Esperados

3. Verificacion operativa minima
