# PLAN_WT-2026-242b

## 1. Contrato Tecnico

Este ticket implementa la capa de contencion para shells Builder huerfanas.
No intenta todavia mejorar la deteccion ni matar procesos desde el launcher.
Su objetivo es evitar que una shell vieja o sin identidad valida contamine el
bus con `HANDOFF_BLOCKED` cuando el ticket ya esta en `READY_FOR_REVIEW` o mas
alla.

Depende de: `WT-2026-242a`

El output minimo es:

- filtrar el caso `stale_builder_round` en la ruta de `agent_controller`;
- no emitir `HANDOFF_BLOCKED` para shells huerfanas cuando el bus ya no espera
  una Builder activa;
- emitir el evento diferenciado y trazable `STALE_BUILDER_ORPHAN` en lugar de
  contaminar el flujo canonico;
- tests focales que fijen ese contrato.

## 3. Cambio Esperado

### Cambio minimo de motor

- La validacion de ronda activa debe seguir detectando `stale_builder_round`.
- La logica de contencion debe aplicarse en ambas rutas:
  - `_handle_mark_ready` (linea ~2726)
  - `_handle_pre_handoff` (linea ~3650)
  Ambas llaman a `_ensure_active_builder_round` y hoy emiten
  `HANDOFF_BLOCKED` directamente sin consultar `bus_state`.
- Cuando el proceso actual falle esa validacion, el controller debe consultar el
  estado derivado del bus antes de emitir un bloqueo.
- Si el ticket esta en `IN_PROGRESS`, el comportamiento actual de bloqueo puede
  mantenerse.
- Si el ticket ya esta en `READY_FOR_REVIEW`, `READY_TO_CLOSE`, `HUMAN_GATE` o
  `COMPLETED`, la shell huerfana no debe emitir `HANDOFF_BLOCKED`.
- En esos estados terminal-adjacent debe emitirse el evento
  `STALE_BUILDER_ORPHAN`.
- `STALE_BUILDER_ORPHAN` no dispara requeue ni transicion de estado.
- `supervisor.py` no requiere cambios en este ticket: el nuevo tipo de evento
  debe comportarse como ruido inerte para los consumidores actuales. Si se
  detecta un consumidor inesperado, ese hallazgo se escala durante la
  implementacion.
- El cambio no debe depender aun de PID persistido; la identidad minima en este
- ticket es `ticket_id + round + bus_state`.

### Resultado observable

- una shell Builder vieja post-success no ensucia el bus con
  `HANDOFF_BLOCKED`;
- `mark-ready` / `pre-handoff` de una ronda valida siguen bloqueando cuando el
  ticket aun esta en `IN_PROGRESS`;
- el ruido tardio queda trazado sin alterar el estado operativo real.

## 4. Tests Esperados

1. Reusar el archivo de tests mas cercano al flujo actual de stale Builder.
   No crear un archivo paralelo salvo necesidad fuerte.

2. Tests gobernantes:
   - caso huerfano con ticket ya en `READY_FOR_REVIEW` no emite
     `HANDOFF_BLOCKED`;
   - caso huerfano con ticket en `READY_FOR_REVIEW` emite
     `STALE_BUILDER_ORPHAN`;
   - caso legitimo en `IN_PROGRESS` conserva el bloqueo actual por
     `stale_builder_round`.
   - `test_blocks_stale_builder_round_before_mark_ready` (test existente) debe
     seguir pasando; cubre la no-regresion del caso `IN_PROGRESS`.

3. Cobertura funcional:
   - demostrar que la ruta de contencion no cambia el contrato canonico de una
     Builder activa;
   - demostrar que el nuevo evento no dispara requeue ni transicion de estado.
