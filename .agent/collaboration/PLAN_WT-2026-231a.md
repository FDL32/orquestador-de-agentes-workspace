# PLAN_WT-2026-231a

## Objetivo
Hacer que `--pre-handoff` commitee de forma determinista los cambios productivos del
`repo_motor` en Modelo B, siempre que esten dentro de `Files Likely Touched` (FLT), y
bloquee mostrando los paths motor-relative fuera de FLT cuando no lo esten.

Este ticket no relaja `mark-ready`: lo alimenta con el commit que hoy falta.

## Root cause verificada
- El cierre exige evidencia de commit con el ID del ticket.
- En Modelo B, el codigo vive en `repo_motor` y el estado operativo vive en
  `repo_destino`.
- El flujo actual ha producido rondas donde `pre-handoff` pasa o bloquea, pero
  `mark-ready` falla por `No commit evidence`.
- La deuda no se arregla con prompt: commitear es fontaneria determinista del harness,
  no una tarea cognitiva del Builder.

## Diseno aprobado

### 1. Reestructurar el guard de motor sucio
La barrera de WT-2026-228a no debe hacer `return 1` antes de la logica de commit cuando
cada cambio productivo del motor esta incluido en FLT.

Nuevo contrato:
- motor sucio dentro de FLT -> commit determinista en `repo_motor`;
- motor sucio fuera de FLT -> bloqueo mostrando los paths motor-relative fuera de FLT;
- sin cambios productivos -> mantener bloqueo de `no implementation evidence`.

### 2. Resolver y clasificar repos
- Declarar `motor_root = _MOTOR_ROOT.resolve()` antes del guard de motor sucio.
  No dejar `motor_root` solo dentro de la rama donde `project_root` no tiene `.git`;
  en Modelo B el destino tambien tiene `.git`.
- Distinguir cambios productivos del `repo_motor` de estado del `repo_destino`.
- No commitear artefactos de colaboracion del destino.
- No crear tags en `repo_destino` en este ticket.

### 3. Normalizacion de paths
Comparar siempre paths normalizados a formato `motor-relative` con `/`:
- FLT: `scripts/destination_context.py`
- git status motor: `scripts/destination_context.py`

No comparar absolutos contra relativos. Una interseccion vacia por normalizacion mala
debe estar cubierta por test.

No usar directamente `parse_files_likely_touched()` para la interseccion del motor:
esa funcion resuelve contra `PROJECT_ROOT`. Para Modelo B, derivar FLT leyendo las
lineas raw del bloque `## Files Likely Touched` y normalizarlas a `motor-relative`.

No usar `get_changed_files()` como fuente de `delivery_changes` del motor: en Modelo B
escanea `PROJECT_ROOT` cuando el destino tiene `.git`. La decision commit-o-bloquea
debe usar cambios productivos del motor (`motor_uncommitted_productive` o helper
motor-aware equivalente).

### 4. Commit automatico en repo_motor
Cuando cada cambio productivo del motor esta dentro de FLT:
1. `git add <paths>` con `cwd=motor_root`
2. `git commit -m "<tipo>(WT-XXXX): <resumen>"`
3. crear/refrescar `checkpoint/review-<ticket>` en `repo_motor`

El mensaje debe incluir el ID exacto del ticket.

El bloque existente de `rel_files` basado en `_path_is_under(Path(f), git_root)` asume
paths absolutos. Para el path de motor, `files_to_stage` son strings `motor-relative`;
pasarlos directamente a `git add` con `cwd=motor_root`.

Cambiar tambien el `cwd` del bloque de tag `checkpoint/review-<ticket>` a
`motor_root`; no basta con cambiar el `cwd` del bloque de commit.

### 5. Hooks que modifican staged files
El harness debe manejar el caso de hook/formatter que modifica archivos staged durante
`git commit`:
- intento 1: add + commit;
- si falla y quedan cambios dentro de FLT, re-add solo esos paths;
- intento 2: commit;
- si vuelve a fallar, bloquear mostrando stdout, stderr y paths motor-relative pendientes;
- nunca re-add fuera de FLT.

## Files Likely Touched
Contrato de resolucion: estos paths son motor-relative. Resolver siempre contra
`repo_motor`, nunca contra `repo_destino`.

- `.agent/agent_controller.py`
- `tests/test_agent_controller.py`
- `tests/test_pre_handoff_guard.py`
- `tests/test_pre_handoff_multirepo.py`

El root local de `repo_motor` es dato efimero de runtime. El launcher lo resuelve desde
`motor_destination_link.json` o desde su propia ubicacion y lo inyecta en el prompt; no
debe guardarse como ruta absoluta en los artefactos canonicos.

Paths prohibidos aunque tengan nombres parecidos en `repo_destino`:
- `.agent/agent_controller.py`
- `.agent/collaboration/work_plan.md`
- `PROJECT.md`

## Tests esperados

### `tests/test_pre_handoff_multirepo.py` (nuevo o equivalente)
1. **motor_dirty_inside_flt_commits_motor**:
   - dos repos git temporales (`repo_motor`, `repo_destino`);
   - cambio productivo en motor dentro de FLT;
   - `--pre-handoff` crea commit en motor con ID del ticket.
2. **motor_dirty_outside_flt_blocks**:
   - cambio productivo fuera de FLT;
   - no commit;
   - salida incluye el path motor-relative fuera de FLT.
3. **empty_round_still_blocks_no_evidence**:
   - sin cambios productivos y sin commit nuevo;
   - mantiene bloqueo por ausencia de commit nuevo y diff productivo.
4. **checkpoint_tag_points_to_delivery_commit**:
   - `checkpoint/review-<ticket>` en motor apunta al commit creado.
5. **normalizes_flt_and_git_paths**:
   - FLT y git status con separadores/base distintos;
   - comparacion converge a paths motor-relative con `/`.
6. **hook_reformat_readd_and_commit**:
   - hook modifica archivo staged dentro de FLT;
   - harness re-add solo paths incluidos en FLT y crea commit limpio en segundo intento.
7. **does_not_use_workspace_changed_files_for_motor_commit**:
   - destino y motor son repos git separados;
   - cambio productivo existe solo en motor;
   - decision de commit detecta el cambio aunque `PROJECT_ROOT` tenga `.git`.
8. **motor_root_defined_when_destination_is_git_repo**:
   - destino tiene `.git`;
   - path Modelo B usa `motor_root` sin `NameError`.
