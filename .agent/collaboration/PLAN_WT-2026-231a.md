# PLAN_WT-2026-231a

## Objetivo
Hacer que `--pre-handoff` commitee de forma determinista los cambios productivos del
`repo_motor` en Modelo B, siempre que esten dentro de `Files Likely Touched` (FLT), y
bloquee con evidencia accionable cuando no lo esten.

Este ticket no relaja `mark-ready`: lo alimenta con el commit que hoy falta.

## Root cause verificada
- El cierre exige evidencia de commit con el ID del ticket.
- En Modelo B, el codigo vive en `repo_motor` y el estado operativo vive en
  `repo_destino`.
- El flujo actual ha producido rondas donde `pre-handoff` pasa o bloquea, pero
  `mark-ready` falla por `No commit evidence`.
- La deuda no se arregla con prompt: commitear es fontaneria determinista del harness,
  no una tarea cognitiva del Builder.

## Diseno aprobado

### 1. Reestructurar el guard de motor sucio
La barrera de WT-2026-228a no debe hacer `return 1` antes de la logica de commit cuando
los cambios productivos del motor estan dentro de scope.

Nuevo contrato:
- motor sucio dentro de FLT -> commit determinista en `repo_motor`;
- motor sucio fuera de FLT -> bloqueo con lista exacta;
- sin cambios productivos -> mantener bloqueo de `no implementation evidence`.

### 2. Resolver y clasificar repos
- Resolver `motor_root`.
- Distinguir cambios productivos del `repo_motor` de estado del `repo_destino`.
- No commitear artefactos de colaboracion del destino.
- No crear tags en `repo_destino` en este ticket.

### 3. Normalizacion de paths
Comparar siempre paths normalizados a formato `motor-relative` con `/`:
- FLT: `scripts/destination_context.py`
- git status motor: `scripts/destination_context.py`

No comparar absolutos contra relativos. Una interseccion vacia por normalizacion mala
debe estar cubierta por test.

### 4. Commit automatico en repo_motor
Cuando todos los cambios productivos del motor estan dentro de FLT:
1. `git add <paths>` con `cwd=motor_root`
2. `git commit -m "<tipo>(WT-XXXX): <resumen>"`
3. crear/refrescar `checkpoint/review-<ticket>` en `repo_motor`

El mensaje debe incluir el ID exacto del ticket.

### 5. Hooks que modifican staged files
El harness debe manejar el caso de hook/formatter que modifica archivos staged durante
`git commit`:
- intento 1: add + commit;
- si falla y quedan cambios dentro de FLT, re-add solo esos paths;
- intento 2: commit;
- si vuelve a fallar, bloquear con stdout/stderr claro;
- nunca re-add fuera de FLT.
